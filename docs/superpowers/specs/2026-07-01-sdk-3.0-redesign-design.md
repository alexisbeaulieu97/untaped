# untaped SDK 3.0 — Ecosystem Consistency Redesign

Date: 2026-07-01 (amended 2026-07-02 after spec review: sequencing trued to the
completed PyPI rollout; full kind-rename inventory; SDK-owned `profile delete` added
to scope; `finish` adoption sweep enumerated; ContextVar fork resolved)
Status: approved design (brainstormed and locked with Alexis)
Target release: untaped 3.0.0, one breaking wave, followed by one ecosystem adoption
wave re-pinning all tools.
Sequencing: the PyPI release rollout is COMPLETE (2026-07-02); the release-tooling
consolidation now lives in this repo and the implementation plan must account for it.
The plan is the next step. The untaped-recipe 0.8.1 hardening and 0.9.0
pack-unification waves are independent (pinned to untaped 2.4.x) and converge with
this wave only at recipe's final re-pin PR.

## Context

Seven independent CLIs (github, jira, awx, workspace, ansible, apple-health, recipe)
build on `untaped.api`. A four-part audit (SDK internals + all local tool repos +
untaped-recipe's dedicated audit) found a healthy SDK (7/10) with three classes of
debt this redesign fixes in one wave:

1. **Documented invariants that are not enforced.** The
   piped-stdin-refuses-without-`--yes` safety rule exists only inside `batch_apply`
   and is opt-in; github `scan clean --repo` deletes with no confirmation (only the
   `--all` path gates on `--yes`), and workspace `remove`/`forget` hand-roll a
   near-verbatim copy of the confirm gate.
   Partial-failure exit codes and `<tool>.<noun>` emit-kind naming are likewise
   convention-only — a full-fleet inventory found 18 live kinds violating the
   intended shape (see Kind validation).
2. **Generic code stranded in tools.** `bounded_map` exists in six near-identical
   copies across four tools; structured-file loading has three independent
   implementations; the SDK keeps an atomic-write+lock facility private to config
   files while recipe built a full transactional writer.
3. **Internal inconsistency.** `ui.py` (584 lines) carries three responsibilities;
   three overlapping rendering entrypoints (one dead); `config/` uses deep DDD while
   `profile/` is flat; three ambient module-global singletons.

Breaking changes are cheap (sole user, no external consumers), so this ships as one
3.0 rather than staged minors.

## Output model (decision record)

**`emit()` is THE output API.** It already shape-dispatches (single → detail view,
sequence → collection), handles empty-state hints, kind tags, and echoing. Tools call
`emit` everywhere; `render_rows` stays exported only for the rare need-the-string case
(e.g. recipe's stderr preview tables); `format_output` and the `output.py` module that
carries it are deleted (zero production callers; its conventions docstring moves to
`docs/`).

**Two-channel model, codified in AGENTS.md:** stdout is data (`emit`, machine-readable,
kind-tagged); stderr is chrome (`UiContext` — messages, progress, prompts, all
quiet/verbose-gated). A typical command body is: do work → `ui` for chrome → `emit`
for data → `finish` for the exit code. Raw `echo(..., err=True)` is reserved for data
intentionally addressed to stderr (diff bodies, previews), never for messages.

**Considered and rejected: return-based output** (commands return typed records, the
framework renders). Elegant for read verbs, but streaming (`--follow`), interactive
destructive flows, and multi-record emissions all still need imperative output — two
coexisting models instead of one, plus a signature change to every command in seven
tools. Its real benefits (no boilerplate, unforgeable conventions) are captured more
cheaply by emit-as-the-API plus 3.0's kind validation and `finish`. Revisit only if
commands ever need library-style programmatic consumption.

## Enforced invariants

- **Destructive-verb contract.** Every destructive verb routes through
  `batch_apply(destructive=True)`: preview → confirm (`--yes`/`-y` to skip; piped
  stdin without `--yes` refuses) → progress → outcome. The generic preview is
  generalized past its delete shape via a `preview` parameter (callable rendering the
  item rows; the current delete-style output remains the default) — workspace
  `remove`/`forget` is the second adopter that motivates the generalized shape.
  The SDK's own mounted `profile delete` is in scope: today it reads raw
  `sys.stdin.isatty()` and lacks the `-y` alias, so the SDK work package brings it
  under the contract (context TTY authority, `-y`, conformance test) — otherwise
  every tool would ship a non-conforming destructive verb via the shared group.
  `config unset` is explicitly OUT: it is symmetric with `config set` (which can
  clobber a value just as silently), single-key, and trivially reversible —
  neither gets a confirm gate.
- **Conformance helper.** `untaped.testing.assert_destructive_contract(...)` runs a
  command twice through `CliInvoker` and asserts (a) piped stdin without `--yes`
  refuses with the standard error, (b) the interactive path prompts and honors
  decline. Each tool adds a one-line conformance test per destructive verb, so the
  invariant survives future changes instead of relying on review.
- **Exit-code helper.** `finish(outcome)` accepts a `BatchOutcome` (or an
  `any_failed`-style bool) and raises `SystemExit(1)` on partial failure, `0`
  otherwise — the per-tool `if outcome.any_failed: raise SystemExit(1)` copies
  disappear, and "3 of 5 deleted → exit 1" becomes an ecosystem guarantee. The
  signature stays bool-accepting precisely so non-`BatchOutcome` failure paths can
  adopt it; the adoption wave sweeps ALL such sites, not just deletes (today: awx
  `_apply_runner`/`_delete`/`_get`/`_launch`; workspace `repo_commands` add+remove,
  `ux_commands`, `lifecycle_commands`, `ops_commands` — the plan re-inventories).
- **Kind validation.** `emit`/`render_rows` validate `kind` against the documented
  shape: `<tool>.<noun>` (lowercase, snake-case noun, optional `.summary` suffix —
  the suffix stays load-bearing for consumers that skip informational records).
  Invalid kinds raise at development time. Adoption renames every non-conforming
  kind — this is a pipe-contract break for downstream scripts, acceptable exactly
  because this is the 3.0 wave; each tool updates its SKILL.md/docs in the same PR.
  The full inventory (18):
  - github: `github.codehit` → `github.code_hit`
  - awx static: `awx.apply-outcome`, `awx.delete-outcome`, `awx.template-usage`,
    `awx.test-case`, `awx.test-result`, `awx.unified-template`,
    `awx.workflow-node` → snake_case equivalents
  - awx dynamic: `awx.job-template`, `awx.workflow-job-template`,
    `awx.execution-environment`, `awx.credential-type`, `awx.instance-group` →
    snake_case equivalents
  - workspace: `workspace.branch-outcome`, `workspace.foreach-outcome`,
    `workspace.sync-outcome` → snake_case equivalents
  - ansible: `ansible.source-status` → `ansible.source_status`
  - apple-health: `health.metric-source` → `health.metric_source`
- **One TTY authority.** `batch_apply` currently reads the real `sys.stdin` while
  `UiContext` reads the injectable context stdin; they can disagree under a test
  harness. Interactivity questions all flow through the context's authority.
- **Scripted prompts in the test harness.** `untaped.testing` gains a scripted
  `PromptBackend` (queue of confirm/select/text answers) injectable through
  `CliInvoker`, so interactive flows are testable at all — the prerequisite for the
  conformance helper.
- **`missing_setting_error` is exported.** It is documented as the standard
  "setting not configured" error but absent from `untaped.api.__all__` today.
- **Flag standard: `-y` aliases `--yes` everywhere** (github currently lacks it).

## New primitives (bubble-ups)

Everything here has two or more independent users or unifies with an existing private
SDK facility. Source references are to today's tool code, which the adoption wave
deletes.

| Primitive | Sources unified | Shape |
|---|---|---|
| `bounded_map(fn, items, *, concurrency, on_each=None)` | 6 copies: github/ansible `_concurrency.py` (identical), 2 awx pools, 2 workspace pools | ansible's canonical form: serial fast-path, cancel-on-error, completion callback. Standalone only — `resolve_each`/`batch_apply` stay sequential; add `parallel=` params later only if adoption shows the need. |
| Atomic + transactional file writes | recipe `infrastructure/file_writer.py` (stage → `os.replace` → reverse rollback) + the SDK's config-private atomic write/lock (`config_file.py`) | `atomic_write(path, content, *, encoding="utf-8", newline="")` and `apply_file_changes(changes)` (multi-file transaction with rollback signaling). Lifted only AFTER recipe's 0.8.1 encoding/newline hardening so the SDK inherits corrected I/O. |
| Structured input loading | jira `read_payload_file`, recipe `load_yaml_mapping_file`, SDK gap | `read_structured_file(path) -> dict` (YAML-or-JSON, `ConfigError` on parse/shape) + `parse_json_pairs(pairs)` (sibling of `parse_kv_pairs` for `KEY=<json>`). |
| Raw-text stdin | jira `_resolve_body` (line-oriented `read_stdin` mangles multi-line bodies) | `read_stdin_text() -> str` (raw, trailing-newline-trimmed) + `resolve_text_input(value, file)` for the flag/file/stdin one-of precedence. |
| Link-header pagination | github `pagination.py` (RFC 5988) | `paginate_link(...)` beside `paginate_offset`/`paginate_pages`; the items-envelope choice stays at the call site. |
| Tool-state collection | workspace `registry_repo.py` + ansible `config_repo.py` (verbatim-comment copies) | `StateCollection(key)` — list/map CRUD over `read_tool_state`/`mutate_tool_state` with upsert/remove/entries and drop-key-when-empty collapse. |
| Unified diff | recipe `diff.py`, zero `difflib` in SDK, `config edit` preview wants it | `unified_diff_text(before, after, *, path)` (patch-compatible `a/`/`b/` headers) + `diff_stats(before, after)` line counts. Takes raw strings — no coupling to file-change models. |

**Reverse move-down:** `paginate_offset` sheds its Atlassian vocabulary — defaults
become neutral (`offset`/`limit`), the `isLast` special case moves behind an explicit
parameter, and jira passes `startAt`/`maxResults` itself. Breaking for jira only; part
of its adoption PR.

## Internal redesign

- **`ui.py` splits**: `ui.py` keeps `UiContext` (prompts/messages/progress);
  `render.py` takes the `Renderer` protocol, `RichTerminalRenderer`, and the ~20
  format helpers. The two halves share only `ThemeSpec`.
- **`config/` flattens to the `profile/` convention** (5 flat files: app, models,
  ports, repository, use_cases). One architecture for both built-in subsystems.
- **Ambient globals become scoped.** `identity._command`, `verbose._verbose`,
  `quiet._quiet` move to `ContextVar`s (decided — not `AppContext`: these are wired
  by `run_tool` at root-option parse time and read by low-level helpers before any
  `AppContext` exists, so riding the context would invert the lifecycle), removing
  the manual reset/environ-snapshot dance in `run.py` and making effects
  invocation-scoped.
- **`config_app.py` splits** command wiring from the `$EDITOR` subprocess, `doctor`
  diagnostics, and interactive prompting helpers.
- Backlog corrections recorded: `CliInvoker` already exists (`untaped.testing`);
  `prompts.py` has no dead helpers — both drop off the deferred list.

## Adoption wave (per tool, one PR each, after 3.0 releases)

- **github**: `scan clean` → `batch_apply(destructive=True)` (its `--repo` path gains
  confirmation for the first time) + conformance test; ~8 collection-emit sites →
  `emit`; `github.codehit` → `github.code_hit`; `-y` alias; local `bounded_map` and
  Link-pagination deleted in favor of SDK.
- **jira**: ~6 collection-emit sites → `emit`; payload/body helpers → SDK versions;
  `paginate_offset` callers pass Atlassian params explicitly.
- **awx**: two hand-rolled pools → `bounded_map`; `job_monitor`'s second paginator
  folds into its existing SDK-backed one; `finish` sweep across
  `_apply_runner`/`_delete`/`_get`/`_launch` + conformance test on `delete`; the 7
  hyphenated kinds rename (see Kind validation).
- **workspace**: `remove`/`forget` → `batch_apply` (hand-rolled `confirm` deleted);
  two pools → `bounded_map`; registry → `StateCollection`; `finish` sweep across
  `repo_commands`/`ux_commands`/`lifecycle_commands`/`ops_commands`; the 3
  hyphenated kinds rename; conformance tests.
- **ansible**: `_concurrency.py` deleted (imports SDK `bounded_map`); alias/source
  repos → `StateCollection`; `ansible.source-status` → `ansible.source_status`.
  Constraint: ansible depends on untaped AND
  untaped-github — bubbled helpers must not create a layering conflict between the
  two (nothing in this spec does; keep it a review checkpoint).
- **apple-health**: re-pin + emit/kind conformance check;
  `health.metric-source` → `health.metric_source` (not otherwise swept locally;
  smallest tool — its adoption PR starts with a quick sweep).
- **recipe**: separate track — after its 0.9 wave, a re-pin PR deletes local
  `file_writer`/`diff.py`/`load_yaml_mapping_file` in favor of the SDK versions (the
  recipe spec's §"Deferred: SDK extraction candidates" is thereby resolved).

## Deferred (recorded so adopter #2 resurfaces them)

Single-user or unproven-shape items, deliberately out: stable cache-path hashing
(github), glob/regex name matcher (github), quote-aware token scanner (github+jira —
two users but tiny and domain-flavored), the GraphQL bisection/HTTP-200-`errors[]`
retry skeleton (github, medium coupling), `retry_idempotent_post` sugar (jira's
per-call `RetryPolicy` is clean enough), `$PAGER` support for large tables, a
workspace/`--all` target-iteration primitive, an FkRef/identity-resolution protocol
(awx/ansible shapes are converging but not ripe), recipe's backup bundle store and
sandboxed-Jinja evaluator core (extract-later per the recipe spec).

## Non-goals

- Return-based command output (rejected above).
- Resurrecting any plugin/registry machinery retired in the SDK-only pivot.
- HTTP-layer redesign (`http.py` is the strongest cluster; only the pagination
  vocabulary change touches it).
- Changing the pipe envelope format (frozen, independently versioned).
- `AppContext.section` cross-model reads stay as-is — cross-tool section reads are a
  deliberate feature of the section registry.

## Versioning

3.0.0 (MAJOR). The breaking set: `format_output`/`output.py` deleted, kind validation
(rejects previously-legal kinds — 18 live kinds rename in adoption, a pipe-contract
break for downstream scripts), `paginate_offset` neutral defaults, `batch_apply`
TTY-authority change (behavioral under embedded harnesses), `profile delete` TTY
authority + `-y`. Everything in "New
primitives" is additive; the internal redesign is invisible behind `untaped.api`.
